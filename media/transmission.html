Media Transmission

Streamed media content, in particular video, is particularly demanding for the underlying transport as it requires both enough bandwidth and low latency.

Well-deployed technologies

To ensure the user experience of streamed media remains resilient in face of changing network conditions, media providers make use of adaptive streaming which selects sources of different quality levels based on the networking environment. Several technologies are available to satisfy this approach (such as MPEG DASH) but Web browsers may not offer native support for them. The Media Stream Extensions (MSE) specification enables Web application developers to create libraries that will consume these different adaptive streaming formats and protocols.

Specifications in progress

When broadcasting popular live events on an IP network, it can be beneficial to the broadcaster to spread the load of video distribution by relying on peer-to-peer transmission among watchers. The WebRTC API enables the establishement and utilization of such a peer-to-peer connection, optimized for audio & video transport.

Exploratory work

To improve the user experience and take advantage of advanced device capabilities when they are available, media providers need to know the decoding (and encoding) capabilities of the user's device. Can the device decode a particular codec at a given resolution, bitrate and framerate? Will the playback be smooth and power efficient? Can the display render HDR and wide color gamut content? The Media Capabilities specification defines an API to expose that information, with a view to replacing the more basic and vague isTypeSupported() and canPlayType() functions defined in HTML.

Media providers also need some mechanism to assess the user's perceived playback quality to alter the quality of content transmitted using adaptive streaming. The Media Playback Quality specification, initially part of MSE, exposes metrics on the number of frames that were displayed or dropped.

Some devices (e.g. TV sets) provide access to non-IP broadcast media; the Tuner API in the TV Control specification brings these non-IP media streams to Web browsers. The API also explores ways to expose and launch broadcast-related applications that get transmitted within the transport stream sometimes. The scope of the API may change in the future though. The work on the API itself has been discontinued while scope, use cases and requirements get refined in the Web and TV Interest Group.

Different media containers, used to transport media over the network, embed different kinds of in-band tracks, which must be mapped onto video, audio and text tracks in HTML5 so that Web applications can access them interoperably across Web browsers. The Sourcing In-band Media Resource Tracks from Media Containers into HTML specification provides such mapping guidelines.

Features not covered by ongoing work

Streaming HTTP media on HTTPs pages
While it is possible to read video/audio content served on HTTP in <audio> and <video> elements served on HTTPs pages (despite the usual mixed content restrictions), this does not extend to the usage of streaming enabled by Media Stream Extensions. There have been early discussions on how this could be solved but the consensus was not to include possible solutions in the current version of the specification.
